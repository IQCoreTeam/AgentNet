# Haptic Intensity Scale Implementation Plan

> **For Claude:** Use the `executing-plans` skill to implement this plan task-by-task.

**Goal:** Tokenize vibration into a 6-level intensity scale and wire haptics into every meaningful flow of the mobile webview (nav, chat, market, on-chain actions, errors).

**Architecture:** All haptics stay in the single existing module `surfaces/webview/src/haptics.ts` (navigator.vibrate wrapper, no-op on desktop). This plan redefines its tokens as an intensity scale, then adds ~15 call sites across components. No native/Kotlin changes — the VIBRATE permission is already in AndroidManifest.xml and `navigator.vibrate` already works in the app WebView.

**Tech Stack:** React + TypeScript (Vite), `navigator.vibrate` Web API.

**Decisions already made with the user (do NOT re-litigate):**
- Card taps (skill/agent cards) DO get a very weak tick.
- Tab/pager switches DO get a tick.
- Failures/deny/blocked-button DO get a double error buzz.
- True amplitude control (Kotlin haptic bridge) is explicitly deferred — duration-based scale only.

**Repo conventions (from user memory — follow strictly):**
- Commit directly to `main`, messages in English, NO Claude co-author/attribution lines.
- A parallel session edits the same working tree: `git add` ONLY the files listed in this plan, never `git add -A`. If `git pull --rebase` hits unstaged parallel files, stash only those, rebase, push, restore.
- No emoji and no em-dash in any user-facing product string.
- Verification: `cd surfaces/webview && npm run build` must succeed. There is no unit-test infra for the webview, and vibration is a device side effect, so verification is build + on-device QA checklist (Task 7). TDD is not applicable here.

## The intensity scale

| Token | Pattern (ms) | Meaning |
|---|---|---|
| `tick` | `8` | Weakest. Selection/navigation: tab change, card tap, sheet/back close |
| `tap` | `12` | Light confirm: drawer, message send, approve, generic buttons |
| `press` | `25` | Something appeared that wants attention: approval card, long-press menu |
| `strong` | `40` | Heavy confirm: on-chain submit buttons, forge approval card |
| `error` | `[30, 60, 30]` | Failure/deny/blocked: double buzz |
| `celebrate` | `[40,30,40,30,40,30,220]` | Success plaque zap (unchanged) |
| `castStart` | `[10, 40, 10]` | Skill starts casting (unchanged, kept as-is) |

`navigator.vibrate` cannot control amplitude, only duration — so "intensity" is duration. 8ms reads as a faint tick, 40ms as a thud.

**Step 1: Replace the token table**

Replace the `export const haptics = { ... }` block so the file body becomes:

```ts
// One place for every WebView haptic. navigator.vibrate is unsupported on desktop
// browsers and many WebViews, so every call is wrapped — a no-op there, a buzz on the
// Android device (VIBRATE permission is in the manifest). navigator.vibrate has no
// amplitude control, so intensity = duration: short reads as a tick, long as a thud.
type Pattern = number | number[];

function buzz(pattern: Pattern): void {
  try { navigator.vibrate?.(pattern); } catch { /* unsupported on this WebView */ }
}

export const haptics = {
  tick: () => buzz(8),                  // selection/nav: tab change, card tap, sheet close
  tap: () => buzz(12),                  // light confirm: drawer, send, approve
  press: () => buzz(25),                // attention: approval card appears, long-press menu
  strong: () => buzz(40),               // heavy confirm: on-chain submits, forge approval
  error: () => buzz([30, 60, 30]),      // failure / deny / blocked action
  castStart: () => buzz([10, 40, 10]),  // light double tap when a skill starts casting
  celebrate: () => buzz([40, 30, 40, 30, 40, 30, 220]), // the success "zap"
};
```

Removed tokens: `longPress` (→ `press`), `forge` (→ `strong`). Kept: `tap`, `castStart`, `celebrate`.
